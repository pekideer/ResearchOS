# 输出语言策略

ResearchOS 面向用户的说明和供人查阅的文档默认使用简体中文，且禁止无必要的中英文混用。

## 基本规则

- 正文句子、段落、表格列名、任务说明、建议、风险说明和验收结果默认使用中文。
- 不得把普通英文词、短语或句子夹在中文表达中；能准确用中文表达的概念必须使用中文。
- 面向用户的报告、简报、读书卡、实时方向文档和汇报材料必须使用自然通用语言，不得使用 `open`、`gate`、`brief`、`core`、`hold` 等机器字段或内部流程词充当正文表达。
- 面向用户解释研究判断时，应写清楚具体含义，例如“这个问题是否仍有研究空间”“是否能形成明确贡献”“以现有数据和时间是否做得出来”，不要写成“开放性门禁”“贡献性门禁”“可行性门禁”。
- 如用户明确要求英文写作、中英双语、英文检索式或英文投稿文本，可按任务需要输出英文；面向用户的解释、风险提醒和验收说明仍优先使用中文。

## 允许保留英文的情况

- 文献原题名、期刊名、会议名、作者名、机构名、DOI、URL 和 Zotero 条目 key。
- 文件名、路径、命令、变量名、配置字段、YAML/CSV/JSON 字段、skill 名、API 名、数据库名和软件名。
- 该文献专门提出的概念、公式名、模型名或不可稳定翻译的术语；首次出现时尽量写成“中文译名（英文原文）”，后续使用中文译名。
- 本领域共同认可且通常以缩写使用的指标、标准或模型，例如 DOI、PMV、PPD、MRT、ADPI、CFD、HVAC、MCDM、TOPSIS、PRISMA。
- 可复制检索式、代码块、命令、正则表达式和数据库查询字段可以保留英文。

## 推荐替换

- `open` 写作“仍有研究空间”“尚未被充分解决”或“问题还没有被现有研究完全回答”；不得在用户正文中直接写 `open`。
- `gate` / “门禁” 写作“判断标准”“检查标准”“推进判断”或“是否值得继续推进”；除文件名、字段名、skill 内部说明外，不在用户正文中使用。
- `gap` 写作“研究缺口”；仅在 skill 名、文件名、字段名或用户明确要求时保留原文。
- `论断` 写作“论断”；`论断-evidence` 可写作“论断-证据”。
- `brief` 写作“简报”；`report` 写作“报告”；`reading card` 写作“读书卡”。
- `live direction` 写作“实时方向文档”；`来源记录` 写作“来源记录”。
- `candidate` 写作“候选”；`route` 写作“路线”；`core` 写作“核心”；`background` 写作“背景”。
- `high / normal / low / hold` 写作“高 / 普通 / 低 / 暂缓”，除非它们是机器字段值。
- `规范化文本` 写作“规范化文本”；`全文缓存` 写作“全文缓存”。

## 学术润色

- 英文论文段落润色时可输出英文修改版。
- 说明、风险提醒和验收结果仍使用中文，除非用户明确要求英文说明。
- 不把普通结果写成空泛宣传语。
- 不改变技术含义。

## 失败处理

- 若普通概念出现无必要英文夹杂，应改写为中文。
- 若用户文档出现内部流程词，应改写为自然解释，并在必要时补一句说明其具体含义。
- 若文件名、路径、命令、字段名或 skill 名被错误翻译，应恢复原文。
- 若中文说明出现空泛拔高，应改为具体、可核查表述。
