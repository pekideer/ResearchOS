# Zotero 写入策略

ResearchOS 默认不写入 Zotero。只有用户明确批准“执行写入”时，才允许进入 Zotero Web API 写入流程。

## 基本边界

- Zotero Local API 不用于写入。
- 写入只能通过 Zotero Web API 执行。
- 真实写入、恢复、清理和项目 collection overlay 写入工具统一隔离在 `tools/zotero/write/`；目录名称描述功能，但其中工具仍全部属于高风险工具。目录存在、脚本存在、dry-run 计划存在或历史批准记录存在，均不构成本次执行授权。
- 一般文献库治理、聚类、矩阵和报告默认只生成 试运行、计划和人工审批清单。
- PRISMA 阅读状态、重要性和计划用途的主状态保存在 ResearchOS 读书卡 YAML 头部 与 `prisma-records.csv`；Zotero 只允许在审批后镜像 `rs:*` 标签。
- ResearchOS 读书卡发布到 Zotero 时只允许创建或版本安全地更新带稳定 `card_id` 的生成笔记；不得写入 annotation，不得覆盖人工修改过的生成笔记。
- 读书卡笔记写后必须验证 note 身份、母条目、标签、结构化内容指纹和版本；验证失败时保留失败审计与回滚计划，不得登记成功映射。
- 本策略不是自动执行授权；每次写入前必须获得用户单独确认。

## 写入前置条件

写入必须同时满足：

1. 已完成试运行。
2. 用户人工确认具体计划。
3. API key 仅来自环境变量。
4. 写入前嗅探当前代理通道，Zotero Web API 请求优先走可用代理。
5. 先执行 金丝雀测试。
6. 小批量执行。
7. 每批保存 执行前/执行后。
8. 每批生成 回滚计划。
9. 失败立即停止。

## 代理规则

- Zotero Web API 写入前必须检查代理通道。
- 代理来源优先级：`ZOTERO_HTTPS_PROXY`、`HTTPS_PROXY`、`HTTP_PROXY`、当前系统代理设置。
- 如果检测到代理，应通过代理访问 `https://api.zotero.org`，除非用户明确要求直连。
- 审计文件只允许记录代理是否启用、代理来源和脱敏后的主机/端口；不得记录代理账号密码、完整代理 URL 或 API key。
- 不把代理配置写入 `.env`、Markdown、报告或脚本硬编码；如需临时指定，只在当前 shell 会话设置环境变量。

## 严禁行为

- 严禁一次性全量更新大量条目。
- 严禁记录真实 API key。
- 严禁记录代理账号密码或完整代理 URL。
- 严禁把 API key 写入 `.env`、Markdown、日志、报告、截图或提示词。
- 严禁删除 Zotero 条目、PDF、笔记或文献集，除非用户明确批准具体删除计划，并另行备份。
- 严禁在 计划未经确认时写入 文献集或标签。

## 金丝雀测试要求

- 先创建或使用一个低风险 金丝雀测试文献集。
- 先对单条低风险 item 做 标签或文献集 测试。
- 保存 金丝雀测试前后状态。
- 用户确认金丝雀测试结果后，才能进入小批量执行。
- 读书卡笔记金丝雀必须限定为一张读书卡和一个母条目，先审查 HTML 预览与具体计划；创建后由用户检查条目归属、排版和链接。

## 回滚要求

- 每批写入前保存 执行前状态。
- 每批写入后保存 执行后状态。
- 回滚计划 必须包含 条目 key、原文献集、原 标签、目标恢复状态和执行限制。
- 新建笔记的回滚只能记录待审批 DELETE 计划，不得自动删除；更新笔记的回滚保存原 note HTML 和更新后版本号。

## 失败处理

- 任一批失败立即停止。
- 输出失败 条目 key、HTTP 状态、错误消息和已执行批次。
- 不自动重试破坏性操作。
