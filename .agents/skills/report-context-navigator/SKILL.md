## 目标

让 ResearchOS 的阶段性汇报不仅说明做了什么，还说明用户现在在哪里、为什么到这里、下一步可以怎么走。该 skill 是 `project-map-builder` 的轻量应用，用于每次任务完成后的汇报收口。

## 输入

- 本次用户请求。
- 本次实际执行内容、生成或修改的文件、检查结果。
- 可选：项目地图、活跃点子、当前课题目录、最近过程记录。
- 可选：用户明确的目标、限制或偏好。

## 工作流

1. 判断用户本次问题想实现的真实目的：获取材料、形成判断、推进选题、整理体系、修复规则、生成报告、准备写作或确认下一步。
2. 汇报执行结果：已完成的动作、生成或修改的文件、实际检查结果、未完成或受限事项。
3. 追加当前位置：当前处于哪个项目、点子、研究阶段或治理阶段；本次任务把路线推进到哪里；仍有哪些未决问题。
4. 追加下一步建议：只给 1-3 条最可能、最有用的下一步；区分必须做、建议做、可暂缓；不把建议写成用户已经决定。
5. 如果用户要求正式报告，可使用 `当前位置`、`下一步建议`、`需要确认` 小节。
6. 如果只是简短任务，压缩成 2-4 句，不强行增加长表。

## 推荐汇报尾部

```markdown
**当前位置**
你现在处在……阶段。本次任务完成了……，把路线从……推进到……。

**下一步建议**
1. 必须先做：……
2. 建议随后做：……
3. 可暂缓：……
```

## 质量规则

- 不替用户做未表达的最终决策。
- 不输出空泛鼓励语。
- 不把可能下一步写成事实。
- 不重复大段已汇报内容。
- 若没有足够项目上下文，明确写“基于本次对话的推断”。
- 涉及 Zotero 写入、文件移动、批量修改时，必须说明审批边界。

## 完成条件

- 汇报已包含完成内容、检查结果、当前位置和下一步建议。
- 下一步建议具体、可执行，并符合当前项目阶段。
- 不确定内容已标注为推断或需要确认。
