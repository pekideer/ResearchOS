# Zotero 阅读状态与项目 collection 语义治理

## 目的

本说明统一 ResearchOS 所有项目的 Zotero 项目 collection、阅读状态标签和读书卡事实源。它不绑定具体课题，不记录任何项目条目或执行结果。

## 核心语义

### 待读

“待读”固定表示尚未生成读书卡。其事实源是集中读书卡索引；Zotero 在审批后可用 `rs:read/todo` 镜像该状态。

阅读状态统一为：

- `rs:read/todo`：尚未生成读书卡；
- `rs:read/initial-card`：已有初步读书卡；
- `rs:read/deep-read`：已完成精读。

“待读”不作为项目用途 collection。需要在 Zotero 中浏览待读文献时，应根据 `rs:read/todo` 建立保存检索。

### 待分配

`00-待分配-triage` 固定表示尚未完成项目相关性与用途分配的临时状态。完成机器分诊、进入明确用途或被排除后即应退出，不要求用户先阅读全文。

### 项目用途

项目 collection 表达文献在具体项目中的用途，例如：

- `01-缺口-gap`
- `02-综述-review`
- `03-引言-intro`
- `04-方法-method`
- `05-实验-experiment`
- `06-参照-benchmark`
- `07-排除-excluded`

默认每条文献保留 1–2 个稳定用途，确有不同作用时可保留 3 个。超过 3 个必须逐条说明必要性。

## 禁止混用

1. 不用项目 collection 表达是否已经阅读。
2. 不把 `rs:read/todo` 理解为项目用途尚未确定。
3. 不把 `00-待分配-triage` 理解为必须等待用户阅读反馈。
4. 不同时创建语义等价的“待读”“待补读”“watchlist”项目 collection。
5. 不因条目退出待分配而推断其已有读书卡。

## 新条目处理

1. 新条目如果用途明确，直接建议必要的稳定项目用途 collection。
2. 项目相关性或用途尚不能判断时，临时建议 `00-待分配-triage`。
3. 尚无读书卡时独立建议 `rs:read/todo`。
4. 生成初步卡或完成精读后更新阅读状态，但不因阅读状态变化自动改变项目用途。

## 增减动作边界

以下动作必须清楚区分：

- 从条目的 `collections` 数组移除一个归属；
- 删除 collection 本身；
- 删除 Zotero 条目、附件或 PDF。

通用写入工具默认保留已有 collection 和标签。只有经过审批的计划显式列入 `remove_collections` 或 `remove_tags` 时，才允许执行对应移除。删除 collection、条目、附件或 PDF 不属于普通治理动作。

## 历史名称迁移

- 新规则、模板和计划只生成 `待分配/triage`。
- 旧“待读/watchlist/待补读”名称只能作为迁移来源或历史审计证据保留。
- 旧阅读标签 `/unread`、`rs:read/unread`、`rs:read/skimmed`、`rs:read/done`、`rs:read/deep` 只作为迁移来源；迁移后由三态标签替代。
- 不覆盖历史执行记录中的原始名称。
- 真实 Zotero 重命名或归属迁移仍需试运行、人工确认、金丝雀测试、分批执行和回滚计划。

## 同步与验收

- Web API 写入完成后，以写后快照作为即时审计证据。
- Zotero 桌面端完成同步后再刷新 ResearchOS 父文档。
- 父文档与写后快照短暂不一致时，不重复写入；先等待同步并复核。
- 历史问题只在迁移当次生成只读差异和回滚材料；迁移完成后不保留针对已消除问题的专用代码入口。
