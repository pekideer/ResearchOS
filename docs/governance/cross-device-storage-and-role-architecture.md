# ResearchOS 跨端存储、角色与生命周期架构

## 1. 目标

ResearchOS 将智能体框架、共享科研语料、项目持久状态和本机运行材料分开治理，使不同终端可以通过 Git、同步盘和项目交接记录恢复工作，同时避免把临时文件、个人配置和具体课题材料混入公开框架仓库。

本架构采用以下已确认决策：

- 每个项目继续使用 `.research/` 保存跨端恢复所需的持久状态。
- 每个本地 ResearchOS Agent Core 使用根目录下被 Git 忽略的 `.researchos/` 保存本机运行材料。
- 采用分域权限；框架、共享语料、具体项目和 Zotero 分别管理写入权。

## 2. 四层存储边界

| 层 | 标准位置 | 是否跨端 | 允许内容 | 禁止内容 |
|---|---|---:|---|---|
| Agent Core | 本地 Git 工作区 | 通过 Git | 规则、skills、流程、模板、工具、测试、通用人读说明 | 真实课题材料、真实语料、密钥、本机配置、运行输出 |
| Shared Corpus | 同步盘 `00_ResearchOS/corpus/`，由本地 Agent Core 通过路径映射或 Junction 访问 | 是 | Zotero 索引快照、规范化全文、集中读书卡、索引、共享语料版本与完整性清单 | Agent 调试日志、任务临时包、未完成预览、工作副本 |
| Project Workspace | 同步盘具体项目目录 | 是 | 正式科研成果和项目专属持久状态 | 可重建缓存、渲染中间件、调试输出、无长期价值的工作副本 |
| Local Runtime | 本地 Agent Core 的 `.researchos/` | 否 | tmp、cache、logs、失败现场、写入前后暂存证据 | 唯一的项目状态副本、唯一的正式审计副本、需要跨端恢复的成果 |

用户主目录 `%USERPROFILE%/.researchos/` 只保存本机路径映射、终端身份、角色配置、代理和私有凭据，不属于 Agent Core 的本地运行目录。

## 3. 项目 `.research/` 契约

项目 `.research/` 是可跨端恢复的持久控制面，推荐结构：

```text
.research/
├── project_manifest.yml
├── run_state.json
├── run-log.jsonl
├── handoff.yml
├── decisions/
├── approvals/
└── audits/
```

允许保存：

- 稳定项目事实、规范输出位置和项目身份。
- 当前阶段、最近完成步骤、未决审批、阻塞项和下一步。
- 最小追加式运行记录。
- 当前项目写入端、交接状态、状态修订号、框架版本和语料快照标识。
- 会影响后续研究判断的重要决策。
- 已批准计划、执行范围和数据最小化后的最终审计凭据。

禁止保存：

- `tmp/`、`temp/`、`cache/`、`debug/`、`preview/`、`render/` 等可重建过程文件。
- 大量原始 API 响应、完整对话、通用模型输出转储或重复全文。
- 只为当前终端服务的绝对路径、代理、密钥和解释器信息。
- 没有恢复、审批、证据或追责价值的中间版本。

## 4. 分域写入角色

### 4.1 Framework Maintainer

- 唯一允许修改、提交和推送 Agent Core 的终端。
- 其他终端只能拉取已发布版本并只读运行框架。
- 最终权限由 Git 远端凭据和分支保护保证，不只依赖脚本或提示词。

### 4.2 Corpus Publisher

- 唯一允许发布共享 SQLite、规范化全文、集中读书卡和索引快照的终端。
- 其他终端只读共享语料；需要性能或稳定性时，可复制到本地 cache 后只读查询。
- SQLite 必须在本地完成写入、关闭连接、校验和生成版本清单后再原子发布；不得由多个终端直接并发写同步盘数据库。

### 4.3 Project Writer

- 每个项目同一时刻只能有一个活动写入端。
- 写入权可以通过项目 `.research/handoff.yml` 在终端之间交接。
- 未持有项目写入权的终端可以读取、分析和提出计划，但不得修改项目成果或项目状态。

### 4.4 Zotero Writer

- Zotero 默认只读。
- 只有被指定的 Zotero 写入端，且在用户批准具体冻结计划后，才能执行 Web API 写入。
- Zotero 写入权不由 Framework Maintainer 或 Project Writer 身份自动获得。

同一终端可以同时承担多个角色，但各角色的授权和审计必须独立判断。

## 5. 审计材料的晋升规则

Agent 运行产生的详细请求、回读、预览和调试材料先进入本机 `.researchos/`。任务完成前，这些材料属于受保护的暂存证据，不得被清理。

任务收束时只把长期需要的最小证据晋升到跨端位置：

- 项目专属决策、审批和审计：项目 `.research/decisions/`、`approvals/`、`audits/`。
- 全库共享语料的版本、哈希、统计和发布审计：共享 `corpus/` 下的专用治理清单位置，具体目录在迁移阶段另行冻结。
- ResearchOS 通用架构和不含真实项目材料的治理结论：Agent Core 的 `docs/governance/`。

晋升完成并核对哈希、引用和恢复能力后，本机详细暂存材料才可进入清理范围。

## 6. 跨端恢复顺序

1. 识别本机终端角色和私有路径配置。
2. follower 终端只执行 Git pull；维护端先确认工作区状态再决定是否更新。
3. 等待同步盘完成，并核对共享语料版本清单。
4. 按唯一恢复链定位项目，读取 manifest、run state、handoff 和最近运行记录。
5. 核对 `handoff.yml` 中的活动写入端、状态修订号、框架 commit 和共享语料快照。
6. 未完成写入权交接时保持项目只读；交接完成后才能修改项目成果和状态。
7. Zotero 写入仍需另行批准，不由换端动作自动恢复授权。

## 7. 当前实施边界

本文件只固化目标契约，不代表已经完成目录迁移、凭据配置、权限强制、清理器部署或项目交接金丝雀。后续实施必须按“盘点、计划、审批、复制、校验、金丝雀、再扩展”的顺序推进；未经明确审批不删除旧文件。
